# AutoQuill 软件修订需求 · 第 01 批

> 本批包含两项相互独立的改动：
> - 需求 ①：推荐页级别的"已答去重"（消除重复采集与下游发布失败）
> - 需求 ②：无障碍树（UI Automation）只读探针（评估是否替换 OCR 读取通道）
>
> ① 可独立实现并先行上线；② 是零写入、零点击的只读探测，其产出用于决定后续是否替换读取层。两者可分开提交、分开验收。

---

## 0. 给执行方的项目背景（自包含，无需额外上下文）

AutoQuill 是一套以 OCR 为感知、以鼠标键盘为动作的"类人"知乎自动创作框架。主流程为：从知乎创作者推荐页采集高赞源回答 → 提炼叙事配方 → 大模型生成新故事 → 发布（以**保存草稿**的形式落地）。整套框架刻意坚持 OCR + 模拟操作，以规避网页端反爬；**本批改动不改变这一总体姿态**。

一次 7 小时实跑（4 轮，每轮生成 66 篇、目标发布 30 篇）暴露出两个问题，本批针对性修复：

1. **重复处理已答问题。** 推荐页刷新慢，已经处理过（已生成草稿）的问题会反复出现在推荐流里。系统当前不加判断地再次采集、再次生成，直到发布环节才撞见按钮已从「写回答」变为「编辑回答」而失败。此失败被当成"无法定位写回答按钮"，触发重试与候补补位，**白白浪费采集/生成时间、消耗候补文章、并拖累后程故事质量**。实测该类失败逐轮递增（8 → 12 → 34 → 33 次）。

2. **OCR 读取不可靠。** 操作层（按钮定位）与信号层（互动数字）都依赖 OCR，错误率偏高。拟评估操作系统无障碍树作为更可靠的读取通道。

## 需求 ① 推荐页级别的"已答去重"

### ①.1 目标

在**尽可能早**的环节（推荐页列表扫描阶段）识别并跳过"已处理过"的问题，从源头消除重复采集、重复生成，以及由此引发的发布失败、候补消耗与后程质量下滑。

### ①.2 关键设计决策（须严格遵守）

**不建立任何持久化的问题 ID 登记表。** 唯一真相源是**页面的实时草稿状态**。理由：本系统交付物永远是草稿，运营者会手动删除自认为不好的草稿；若用问题 ID 登记"已答"，被手动删除的草稿在登记表中仍会残留为"已答"，导致登记表与人工干预**漂移失真**。而页面状态会随草稿删除**自动回退**（删除草稿后该问题重新显示为"可答"，系统应当重新处理它——这正是期望行为）。因此**全部判定都基于页面当前状态，不写任何跨会话的已答记录**。

### ①.3 当前行为

`collect_materials_batch`（见 `workflows/zhihu.py`）扫描推荐页时不判断草稿状态，直接点入问题采集；发布阶段（`publish`）才发现「编辑回答」并失败，且该失败与"真正找不到写回答按钮"被混为一类，触发重试 + 候补补位。

### ①.4 目标行为（两层）

**A 层 —— 列表级快筛（主路径）**

推荐页每一行的「操作」列包含一个左侧动作按钮（位于始终存在的蓝色「写回答」按钮左侧）：

- 文案为 **`已加草稿`**（灰色不可点）→ 该问题**已处理过**：本行**整行跳过**——不点入、不采集、不提炼配方、不生成故事。记一条 `已跳过（已加草稿）` 的 INFO 日志。
- 文案为 **`稍后答`**（蓝色描边可点）→ **全新问题**：按现有逻辑正常采集。

定位提示（不强制实现方式）：目标文案在最右「操作」列中、蓝色「写回答」按钮的**左邻**。可沿用现有的 OCR 区域识别思路在该行右段定位。须注意不要把问题标题旁的 `飙升` 等红色标签误判为动作按钮。

**B 层 —— 详情页兜底**

对通过 A 层、确实点入的问题，在发布前定位按钮时：

- 若页面出现 **`编辑回答`**（或明确不存在 `写回答`）→ 判定为**已答**（A 层因列表状态滞后而漏判）→ **干净跳过该条**：不发布、**不计为发布失败、不消耗候补文章**，直接处理下一条。记 `已跳过（已答/编辑回答）` 日志，与真正的发布失败区分开。

**关键区分（执行方务必正确实现）：** 当前实现把「编辑回答」与「无法定位写回答」混为一类失败。修订后，**仅当既找不到 `写回答` 也找不到 `编辑回答`（页面真正异常/丢失）时**，才算真实失败并走原有的重试/候补逻辑；**「编辑回答」存在 = 已答 = 干净跳过，不是失败**。

### ①.5 边界与异常处理

- **列表行动作按钮 OCR 读不出来**：默认**不跳过**（按"全新"处理，照常点入），交由 B 层兜底。原则是"采集侧失败放行、发布侧失败保守"——错跳一个新问题（损失产出）比多点一次（由 B 层廉价拦截）更糟。
- **手动删除草稿导致状态回退**：无需特殊处理。该行重新显示 `稍后答`、被重新处理，即为预期的自我修正行为。
- A 层与 B 层是冗余的两道关卡，任一独立生效，组合使用。

### ①.6 验收标准

1. 推荐页混合存在 `已加草稿` 与 `稍后答` 行时，系统**只对 `稍后答` 行采集与生成**；`已加草稿` 行产生"已跳过"日志且零生成。
2. 由「编辑回答」引起的 `无法定位写回答按钮` ERROR 降至约 0；任何在发布阶段遇到的「编辑回答」被记为干净跳过，**不产生 ERROR、不消耗候补文章**。
3. 逐轮的发布失败次数不再随轮次递增（对照实跑 8→12→34→33 的趋势消失）。
4. 代码中**不存在**任何持久化问题 ID / 已答记录的写入。

### ①.7 不在本需求范围内

不改动生成、评分、配方提炼逻辑；不涉及自进化层。

## 需求 ② 无障碍树（UI Automation）只读探针

### ②.1 目标

用实证手段判断：Windows 无障碍树（UI Automation）能否比 OCR 更可靠地读取我们依赖的页面元素（按钮、标题、互动数字），**但本阶段不替换 OCR**。这是一个**只读诊断探针**，其输出用于驱动下一步架构决策（是否、以及在哪些字段上改用无障碍树）。

### ②.2 给执行方的原理说明

无障碍树是操作系统为读屏软件提供的客户端接口（Windows 上为 UI Automation），把当前窗口控件以结构化文本暴露（每个元素含：控件类型 / 名称文本 / 屏幕坐标）。浏览器会把网页内容映射进该树。**它纯客户端、不发网络请求、不带 webdriver/CDP 指纹，服务器侧无感**——读取的是"已渲染像素背后的文本"，与"另起请求抓数据"是两回事，因此不触发反爬。预期它能精确给出按钮与标题文本，免去 OCR 的识别概率问题。

**已知前提**：运营者已实测确认知乎相关数字**未使用字体混淆**（可正常显示与复制），故互动数字预计可经无障碍树取得；本探针即用于确认覆盖范围。

### ②.3 形态

在 `main.py` 中新增一个**调试分支**，与现有 `--debug-ocr-region` / `--test-ocr` 同级（命名示例 `--probe-a11y`）。该分支**不点击、不发布、不生成**，仅读取并转储。

### ②.4 行为规格

- 复用现有窗口聚焦工具，作用于已打开的自动化 Edge 窗口。
- 覆盖两个场景（可分别触发或顺序执行）：
  1. **推荐页**（创作者推荐问题页）。
  2. **某个问题详情页**（推荐通过 URL 参数传入，或对"当前已打开页面"操作，以保持零点击）。
- 对当前窗口遍历无障碍树，**逐元素转储**：控件类型、名称/值文本、外接矩形（屏幕坐标）。输出同时打印到控制台并写入 `logs/` 下带时间戳的文件，便于事后查阅；格式为可读的缩进树或带层级的扁平列表。
- 末尾附一段**聚焦摘要**，专门报告：
  - 名称匹配 `写回答 / 稍后答 / 已加草稿 / 编辑回答` 的按钮是否找到、及其坐标。
  - 问题标题文本元素。
  - 详情页上的 `赞同/点赞` 数值，以及（若存在）`评论/收藏`——**原样打印取到的字符串**，以便人工核对是否为真值。

### ②.5 技术提示（非强制实现细节）

Windows 上可用 `uiautomation` 或 `pywinauto`（UIA 后端）等成熟库实现；**只能依赖客户端无障碍 API，不得发起网络请求、不得使用 webdriver/CDP**，以保持"类人、不爬取"的姿态。Edge 的网页内容在无障碍树中暴露；深层网页内容可能需在辅助客户端挂载时才激活，执行方需留意。

### ②.6 决策矩阵（探针输出须能填满此表）

| 待读取对象 | 探针须回答 | 结论 → 决策 |
|---|---|---|
| 按钮（写回答/稍后答/已加草稿/编辑回答） | 能否取到名称正确的按钮元素及坐标 | 能 → 操作层/去重层可改用无障碍树，消除定位失败 |
| 问题标题 / 题材 / 飙升标 | 是否为可读真实文本 | 能 → 列表解析可改用无障碍树，更稳 |
| 点赞/评论/收藏等数字 | 取到的是真值、还是空/异常 | 真值 → 信号层可改用无障碍树（对未来自进化校准是大幅可靠性提升） |

### ②.7 验收标准

1. 对推荐页运行探针，输出能清晰显示四种动作按钮文案与问题标题是否作为带坐标的命名元素存在。
2. 对问题详情页运行探针，输出能显示 `赞同` 数是否可作为文本取得，且取到的字符串是否等于页面可见数字。
3. 探针全程**无点击、无额外网络请求、不创建草稿、完全只读**。
4. 输出足以让架构方填满 §②.6 决策矩阵，并就 (a) 操作、(b) 列表解析、(c) 数字 三个方向**各自独立**决定是否采用无障碍树。

### ②.8 不在本需求范围内

不实际替换 OCR；不经无障碍树做任何写入/点击；不做性能优化。
